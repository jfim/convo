# Convo ‚Äî Claude Code Conversation Renderer

**Date:** 2026-05-28
**Status:** Approved design, pre-implementation

## Purpose

A desktop application that renders a Claude Code conversation transcript into a
readable view. A conversation is addressed by a `convo://` URL, e.g.:

```
convo://claude-code/-home-jfim-projects-alchimiste/e83d06a0-1c87-4dd3-903c-fd944b0df140
```

which resolves to and renders:

```
~/.claude/projects/-home-jfim-projects-alchimiste/e83d06a0-1c87-4dd3-903c-fd944b0df140.jsonl
```

The scope is deliberately narrow: faithfully render one conversation. It is not a
chat client, an editor, or a sync tool.

## Goals (v1)

- Register the `convo://` URL scheme with the OS so links open the app and render
  the referenced conversation.
- Support a turn anchor in the URL fragment (`#<uuid>`) that scrolls the matching
  event into view and briefly highlights it.
- Render **full fidelity**: user/assistant text, thinking blocks (collapsible),
  tool calls and their results, attachments, system events, and PR links.
- Never silently drop data: unknown event/block kinds and malformed lines render
  as visible, clearly-marked nodes.

## Non-goals (v1)

- **Live tail.** Not built in v1, but the architecture must allow appending new
  events later without rewriting existing DOM (see Architecture).
- **Conversation picker / browser.** Single conversation per launch. A picker over
  `~/.claude/projects` (and statistics) is a likely future addition; the
  parse/resolve layers are kept independent of "which conversation" so this can be
  added without rework.
- **Mutation of conversations.** Read-only. The only future write-adjacent concern
  is appending newly-observed events to the view, never editing the file.

## Input format (observed)

The `.jsonl` files contain one JSON object per line. Findings from inspecting a
representative 1,805-line file:

- Top-level `type` values seen: `user`, `assistant`, `system`, `attachment`,
  `queue-operation`, `last-prompt`, `pr-link`.
- Events thread via `uuid` / `parentUuid`. Common envelope fields include
  `sessionId`, `timestamp`, `version`, `userType`, `isSidechain`, `gitBranch`,
  `entrypoint`, `cwd`.
- **Assistant** messages: `message.content` is an array of blocks with
  `type` in {`text`, `thinking`, `tool_use`}.
- **User** messages: `message.content` is **either** a string (a prompt) **or** an
  array (containing `tool_result` blocks, possibly with image content). Tool-result
  blocks carry `tool_use_id`, `content` (string or array), `is_error`.
- **System** messages: `subtype` in {`stop_hook_summary`, `api_error`}, with
  `level`, hook fields, etc.
- **Attachment** events: hook outputs (command, stdout, stderr, exitCode,
  hookName, hookEvent) and file-diff summaries (addedLines, addedNames,
  removedNames).
- `toolUseResult` carries structured tool output; `pr-link` events carry
  `prUrl` / `prRepository` / `prNumber`.

The schema is large, partly optional, and evolving. The design treats it as such:
lenient decoding with explicit fallbacks rather than a rigid struct.

## Architecture

Chosen approach: **Rust parses the JSONL into typed events; the frontend renders
them incrementally.** (Alternatives considered: frontend-side parsing in TS, and
Rust rendering HTML strings. Rejected ‚Äî the first pushes fragile schema parsing
into an untyped language; the second loses component reactivity needed for
collapsibles and future live-tail. Approach may be revisited if the typed boundary
proves more friction than value.)

Stack: **Tauri 2.x + Svelte + TypeScript (Vite)**, Rust backend.

Tauri/Rust was chosen over an Elixir LiveView + ElixirKit alternative explicitly:
for an "open a link, render one file" tool, a native binary + system webview has
far lower cold-start and idle overhead than booting a BEAM VM + Phoenix endpoint,
and the append-only single-file tail is the degenerate case LiveView's
state-diffing model would be overkill for.

Shared types are **auto-generated from the Rust structs** via `tauri-specta`,
producing `src/lib/bindings.ts` (types + typed command wrappers). The frontend and
backend cannot drift. This auto-generation is a load-bearing requirement of the
chosen approach.

### Data model (`model.rs`)

- An envelope struct carries the common fields (`uuid`, `parent_uuid`,
  `session_id`, `timestamp`, `cwd`, `git_branch`, ‚Ä¶), all non-guaranteed fields as
  `Option<T>`, plus a typed payload.
- The event kind is a `#[serde(tag = "type", rename_all = "kebab-case")]` enum with
  variants for each observed type and `#[serde(other)] Unknown` so unrecognized
  future types decode rather than fail.
- Assistant content decodes into `Vec<ContentBlock>`, an enum over
  `Text` / `Thinking` / `ToolUse` with an `Unknown { raw: serde_json::Value }`
  fallback variant.
- User content is an untagged enum: `Prompt(String)` vs `Blocks(Vec<ContentBlock>)`,
  handling the string-or-array shape.
- All public model types derive `serde::{Serialize, Deserialize}` and
  `specta::Type` so they cross the boundary as generated TS.

### Data flow

1. App is launched with a `convo://claude-code/<encoded-project>/<uuid>` URL
   (deep link), optionally with a `#<uuid>` fragment. In dev, a file path may be
   passed directly.
2. `resolver.rs` maps the URL to
   `~/.claude/projects/<encoded-project>/<uuid>.jsonl`, validating the resolved
   path stays within the projects root (reject path traversal).
3. `commands::load_conversation` invokes `parser.rs` and returns an ordered
   `Vec<ConversationEvent>` (parent/child resolved) across the typed boundary,
   along with the optional anchor uuid.
4. `Conversation.svelte` renders one keyed component per event.

### Deep link

`tauri-plugin-deep-link` registers the `convo://` scheme and handles per-OS
plumbing. v1 handles the `claude-code` host; `resolver.rs` is written so other
hosts/sources can be added without touching the parser.

### Rendering & append-friendliness

- `Conversation.svelte` renders `{#each events as e (e.uuid)}`, keying by `uuid`.
  Each event maps to a component by kind; assistant content blocks map to block
  components.
- Because the list is keyed by `uuid`, the future live-tail path is purely
  additive: push new events onto the array and only new nodes mount; existing DOM
  is untouched.
- Thinking blocks render collapsed by default. Tool-use and tool-result render
  together, matched by `tool_use_id`.
- Unknown event/block kinds and parse-error nodes render via `UnknownNode.svelte`
  as a visible, muted block showing the raw JSON ‚Äî nothing is silently dropped.
- Turn anchor: if the launch URL carried `#<uuid>`, scroll that node into view and
  briefly highlight it after render.

### Error handling

- **Bad URL / path traversal / missing file:** `resolver.rs` returns a typed error;
  the app shows a clear error screen and does not panic.
- **Malformed JSONL line:** the parser decodes line-by-line; a failing line becomes
  a `ParseError` event carrying the raw text and line number, rendered inline. One
  bad line never aborts the conversation.
- **Unknown fields/types:** absorbed by `Option` and the `Unknown` enum variants.
  Boundary error types are part of the generated TS so the frontend handles them
  explicitly.

### Testing

- **Rust unit tests** are the core. `parser.rs` is tested against fixture `.jsonl`
  snippets ‚Äî one fixture per event/block kind, plus malformed-line and unknown-type
  fixtures. `resolver.rs` is tested for URL‚Üípath mapping including
  traversal-rejection cases.
- **Fixtures** are small hand-authored `.jsonl` files in `src-tauri/tests/fixtures/`,
  never real conversations (avoids committing user data).
- Frontend component tests are optional for v1, since the logic concentrates in
  Rust.

### Tooling

- `rust-toolchain.toml` pins the stable toolchain plus `rustfmt` and `clippy`
  components.
- `rustfmt.toml` minimal (pins edition style).
- `.tool-versions` (asdf): `nodejs`, `just`.
- `justfile` recipes: `dev`, `fmt`, `lint` (`clippy --all-targets -- -D warnings`),
  `test`, `check` (fmt + lint + test), `build`.
- `clippy` is denied-on-warnings in `check` and CI.

## Open questions / future work

- Live tail: file watcher emitting new events to the keyed list (architecture
  already supports it).
- Conversation picker over `~/.claude/projects`, and conversation statistics.
- Rendering of image content blocks inside tool results (decode/display strategy).
